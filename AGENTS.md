# Codex 项目规则

这个项目的目标是把 OpenCLI 采集到的 AI 信息源，整理成稳定、易读、有洞察的 AI 日报。

## 启动时优先检查

每次进入本项目，先检查最新的 AI Radar 采集结果：

```bash
npm run ai:prepare-brief
```

如果脚本提示存在 `analysisStatus: pending` 的运行目录，请读取该目录里的：

- `analysis-input.md`
- `basic-report.md`
- `raw/all-results.json`
- `templates/ai-radar-daily.md`

然后生成：

```text
runs/ai-radar/<run-id>/ai-brief.md
```

生成完成后运行：

```bash
npm run ai:mark-brief-done -- runs/ai-radar/<run-id>
```

## 资料库归档要求

采集层和分析层产物分开归档：

- OpenCLI 采集层 Markdown 归档到：`opencli 数据爬取库/YYYY-MM-DD-数据爬取.md`
- Codex 生成的 AI 日报归档到：`Codex AI日报库/YYYY-MM-DD-AI 日报.md`

`runs/ai-radar/<run-id>/` 仍然保留原始 JSON、筛选结果和可追溯运行文件。

## 日报写作要求

整体原则：先结论后细节，避免新闻堆砌；输出中文，表达清楚、具体、少套话。

| 项目 | 要求 |
| --- | --- |
| 栏目 | 每条日报只保留：发生了什么、为什么重要、我们该关注什么、今日术语解释、原始来源；不再写配图/媒体栏目。 |
| 禁用栏目 | 不再写“一句话总结”“背后反映的趋势”“对我们有什么用”“行动建议”“选题建议”这些独立栏目。 |
| 发生了什么 | 用一条列表项写，可以使用正常专业术语，要相对全面地交代事件全貌，但必须凝练，不能冗余。 |
| 为什么重要 | 用两个列表项分开写：第一条说明这件事为什么重要，第二条说明它反映了什么趋势；不要在列表项后再加独立小标题。 |
| 我们该关注什么 | 用一条列表项写，说明它对我们判断方向、设计工作流、做内容或选择工具的启发。 |
| 今日术语解释 | 必须用通俗语言解释关键概念，可以给简单例子或打比方。 |
| 数量与新鲜度 | 日报必须生成 25 条，且只允许使用采集当天或前一天的新鲜内容；不要用更早内容凑数。 |
| 事实边界 | 只基于采集数据和可见来源，不要编造未在来源里出现的事实。 |
| 弱确认来源 | 如果信息源只是首页快照或候选链接，必须标注“需进一步确认”。 |
